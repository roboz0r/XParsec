# 2. Building the lexer: tokens, strings, fused operators

## Hook

F#'s lexer looks simple until you open the spec. Nested block comments. Interpolated strings whose holes contain further F# source. Fused multi-character operators whose shape only disambiguates at parse time. Numeric literal forms for every integer size in .NET plus BigInteger, decimal, and native-sized integer. And every bit of source text has to come out with a position, because every diagnostic downstream needs one.

This post is about the *pure* lexer — `Token.fs` and `Lexing.fs`. It doesn't cover preprocessor directive handling, virtual tokens for layout, or the offside rule. Those live one layer up, in the lexical filter; that's the next post.

## What this post covers

- **Tokens as `(Kind ||| Precedence)`.** The single most leveraged design decision in the codebase. `OpGreaterThan`, `OpLessThan`, `OpInequality`, `OpLessThanOrEqual`, `OpGreaterThanOrEqual` *all* equal `(KindOperator ||| Precedence.ComparisonAndBitwise)` — numerically indistinguishable. That's deliberate: Pratt parsing wants one precedence, and the spelling is recovered from the token's source span when needed. Show the `Token` enum in `Token.fs` and the `TokenInfo.isOperator` / `TokenInfo.canBePrefix` helpers.
- **Sixteen bits per token kind.** A design goal I set early and never compromised: the whole `PositionedToken` has to fit in 64 bits. That's `uint16 Kind` plus `uint32 StartIndex` with room left over. To make 16 bits enough for the kind, the enum had to carry *rich* metadata — precedence level, operator/keyword flags, the `IsVirtual` bit — all packed into those same 16 bits via bitwise OR. Early drafts had 32-bit kinds with separate flag fields; each pass compressed the layout further. The progression is visible in `d99bb47 Reworking Token definition`, `1a28536 Remove ParseSuccess struct in FSharp parser`, and `0d9a31b Reworking Token definition`. Every time I was tempted to add a new field, I compressed an existing one instead. Why: an F# file is easily 50k tokens. At 16 bytes per token that's 800 KB of random-access cache-unfriendly data; at 8 bytes it's 400 KB and fits comfortably in L2.
- **No length field.** Derived, not stored. Any token's length is `tokens[i+1].StartIndex - tokens[i].StartIndex`. This only works because *nothing is missing* between adjacent tokens — which is only true because of the next bullet.
- **Trivia stays in the lexed output.** Whitespace, newlines, line comments, block comments all emit tokens. The token stream is a *complete* cover of the source text with no gaps, which is what makes the "length from next StartIndex" trick sound. But the bigger payoff is elsewhere: F#'s grammar has adjacency-sensitive rules — high-precedence application (`f x.y` vs `f x .y`), indexer access (`f[i]`), type application (`f<T>` only when no space before `<`), measures, and several others. With trivia tokens in the stream, "are these two tokens adjacent?" is answered by "is the previous array slot a non-trivia token?" — see `isPrevTokenNonTrivia` in `ParsingHelpers.fs`. Without trivia in the stream the same question requires carrying column deltas through the parser. And as a bonus: formatters, refactoring tools, and syntax highlighters all need trivia anyway. The lexical filter (post 3) is the one place trivia is hidden — from the parsers' point of view — so no consumer has to think about it unless it wants to.
- **Keywords and numbers first.** The oldest commits (`dce6227 Add tests for keyword and numeric parsers`, `83c06d6 Add tests for lexing block comments`, `6fd50c3 Fully lex some basic files`) are tests for the unglamorous parts of any lexer. Walk through `e89a38a Fix lexing of floating point literals` as a case study — the kind of bug that hand-written tests miss and corpus testing (post 10) finds immediately.
- **String sub-modes.** `"..."`, `@"..."`, `"""..."""`, `$"..."`, `$"""..."""`, `$@"..."`, byte arrays `"..."B`. Each has a different escape rule. The lexer runs a state machine; the design of that machine is covered in depth in post 7. Here, just show the set of token kinds a string expands into: `StringOpen`, `StringFragment`, `EscapeSequence`, `FormatPlaceholder`, `InterpolatedExpressionOpen`, `InterpolatedExpressionClose`, `StringClose`.
- **All strings are fragments.** Commit `17172d4 Parse all string literals as fragments`. Originally a simple `"hello"` was one `StringLiteral` token; after corpus feedback it became `StringOpen StringFragment StringClose`. That shift unified the plain-string and interpolated-string shapes so downstream parsers stopped caring about the difference.
- **Unterminated strings and identifiers.** `bb959c3 Handle unterminated interpolated strings`, plus `UnterminatedStringLiteral`, `UnterminatedBacktickedIdentifier`. Rather than fail the lex, emit a distinct token kind and keep going. Diagnostics are attached at parse time. A single broken string at the top of a file shouldn't destroy the lexer's ability to highlight errors on line 200.
- **Fused operators as produced.** The lexer *fuses* `>]` into a single `KWRAttrBracket`, `^-` into a fused operator, and so on — because unfused, these are genuinely ambiguous with whitespace-separated pairs. The lexer's job ends at fusing; the question of whether a given occurrence *should have been* two tokens is answered one layer up by the filter and its `SplitRAttrBracket` / `SplitPowerMinus` state bits. Post 3 shows the unfusing.
- **Preprocessor and warn directives as tokens.** The lexer produces `IfDirective`, `ElseDirective`, `EndIfDirective`, `NoWarnDirective`, `WarnOnDirective` as first-class token kinds. It doesn't evaluate them. Evaluation is the filter layer's job (post 3).
- **Virtual token bit.** `TokenRepresentation.IsVirtual` is a high-order bit on the token kind. The lexer never sets it; the filter layer synthesizes virtual tokens by OR-ing the bit into an otherwise normal kind. Included here because the bit-encoding is part of the token type.
- **`(*)` is a function identifier.** Commit `22a8fc7 Fix lexing "(*)"`. A one-paragraph story about the cost of greedy comment-open matching. The lexer has to prefer the `(` `*` `)` reading over the `(*` comment-open when the closing paren is adjacent.
- **Line-start index.** `state.Lexed.LineStarts` is a parallel array giving the token index of each line's first token. It's the substrate for column lookups (used by every offside check) and for the preprocessor's "scan this directive's line" logic.

## Takeaway

Four decisions carried the weight. **Precedence packed into the token kind** simplified the Pratt parser. **Fused operators without a commitment on how they'll be used** pushed the split decision up into the filter layer, where it belongs. **Trivia retained in the stream** turned adjacency checks — which F# depends on heavily — into a single array lookup, and made tooling use the same data structure as parsing. **Eight bytes per token** was a constraint that forced every other design choice to be tight: the length field disappeared, the flags compressed, and nothing grew without something else shrinking. None of these would read as load-bearing on their own. Together they set the ceiling for the entire rest of the parser.
